# 아키텍처 개요

## TL;DR (10줄 이내)
- Clean Architecture를 기본 구조로 사용한다.  
- 의존성은 항상 안쪽 계층으로만 향한다.  
- 구조 상세는 `docs/architecture/structure.md`에 기록한다.  

## 사용
- 레벨: L0 (항상 읽기)  
- 읽기: 세션 시작 시 또는 구조적 판단이 필요할 때.  
- 갱신: 아키텍처 원칙/경계/의존성 규칙이 변경되었을 때.  
## 목표
- 도메인 규칙을 외부 변화로부터 보호한다.  
- 유지보수와 테스트 용이성을 높인다.  

## 계층
- Domain: 엔티티, 값 객체, 도메인 서비스  
- Application: 유스케이스, 애플리케이션 서비스  
- Interface: 컨트롤러, 프레젠터, DTO  
- Infrastructure: DB, 외부 API, 메시징 등 구현  
- Entrypoints: HTTP/CLI 등 진입 지점  

## 의존성 규칙
- 바깥 계층은 안쪽 계층에만 의존한다.  
- 도메인은 어떤 프레임워크나 인프라에도 의존하지 않는다.  
- 인터페이스/인프라 구현은 도메인/애플리케이션을 침범하지 않는다.  

## 경계 규칙
- 입력은 인터페이스 계층에서 정제된 DTO로 전달한다.  
- 출력은 포트(인터페이스)로 정의하고 구현은 인프라에서 제공한다.  
- 공통 유틸은 계층 경계를 침범하지 않는 위치에 둔다.  

## 테스트 전략(초기)
- Domain/Application은 단위 테스트 중심으로 구성한다.  
- Interface/Infrastructure는 통합 테스트로 보완한다.  

## 구조 변경 시
- 변경된 경로와 목적을 이 문서에 즉시 반영한다.
